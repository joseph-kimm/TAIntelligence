Document databases and relational databases differ in several key ways:

* **Data model and schema** – A document database stores data as self‑contained JSON (or XML) documents that can have arbitrarily nested structures, and the database does not enforce a fixed schema.  The schema is implicit and interpreted only when the data is read (schema‑on‑read) <<chunk_4,chunk_10>>.  In contrast, a relational database uses tables with rows and columns and enforces a strict schema at write time (schema‑on‑write) <<chunk_10>>.

* **Query language** – Relational systems are queried with SQL, a declarative language that provides powerful join operations and set‑based processing.  Document systems use a variety of APIs; some expose only key‑value access, while others provide secondary indexes and richer query languages such as MongoDB’s aggregation pipeline, JSONPath, or XQuery for XML <<chunk_2,chunk_8>>.

* **Joins and relationships** – Relational databases natively support joins and many‑to‑many relationships, making it easy to reference any row directly by its primary key.  Document databases can represent relationships by embedding related data or by storing reference IDs, but they cannot directly refer to a nested item; joins must be emulated (e.g., with $lookup in MongoDB) or handled in application code <<chunk_4,chunk_9>>.

* **Locality and performance** – Because a whole document is stored together, retrieving an entire document often requires a single read, which can be faster when the application needs most of the data at once.  However, updating a document usually rewrites the whole document, so frequent small updates are less efficient.  Relational tables split data across multiple rows, requiring multiple index lookups to assemble a full object, which can involve more disk seeks <<chunk_8>>.

* **Flexibility vs. structure** – Document databases excel when the data naturally fits a tree‑like, one‑to‑many structure and when schema flexibility is important (e.g., evolving fields over time).  Relational databases excel when strong consistency, complex joins, and well‑defined relationships are required <<chunk_9>>.

* **Hybrid trends** – Modern relational systems have added JSON types and the ability to index inside documents, while some document databases have added join operators and secondary indexes, blurring the line between the two models <<chunk_1>>.

Large‑language models can be extended with **tool‑calling** (i.e., invoking external APIs or software) in several ways that have been explored in recent research:

1. **Built‑in plugins / API‑call interfaces** – Leading LLM providers (e.g., OpenAI’s GPT‑4) have added a “function‑call” or plugin capability that lets the model emit a structured API request instead of plain text. The model can then send that request to a real‑world service (flight search, weather, translation, etc.) and incorporate the returned result into its next response <<chunk_1,chunk_9>>.

2. **Explicit tool‑use prompting** – By providing the model with a description of the available tools (e.g., “web‑browser, calculator, Python interpreter”) in the prompt, the model can be guided to generate calls to those tools when needed. Early work demonstrated this with a small, well‑documented set of tools such as web‑browsing, calculators, translation systems, and Python interpreters <<chunk_2>>.

3. **Tool‑specific instruction‑tuning datasets** – Projects like **ToolBench** automatically construct large instruction‑tuning corpora (hundreds of thousands of examples) that pair natural‑language requests with the exact API call (including required parameters). Training on such data teaches the model to map user intents to concrete tool invocations across thousands of real‑world REST APIs <<chunk_3,chunk_7>>.

4. **Retriever‑aware training** – When the model must use up‑to‑date API documentation, the training data appends a retrieved documentation snippet to the user prompt (e.g., “Use this API documentation for reference: \<JSON\>”). This teaches the model to parse the second half of the prompt, select the correct API, and fill in the parameters, which improves test‑time adaptation and reduces hallucinations <<chunk_5>>.

5. **Multi‑tool composition** – Instead of calling a single API, the model can be trained to generate **multi‑tool** queries that combine several services in one workflow (e.g., fetch cat facts, retrieve NBA news, and book a hotel). The instruction‑generation pipeline explicitly samples combinations of APIs to create such multi‑tool scenarios, encouraging the model to reason across tools and perform multi‑step reasoning <<chunk_7,chunk_10>>.

6. **Evaluation frameworks** – Benchmarks such as **ToolBench**, **APIBench**, and **Toolqa** provide real‑world API calls, responses, and multi‑step reasoning traces. These datasets are used to evaluate how well a model can select the right tool, generate correct request payloads, and handle the returned data, guiding further improvements in tool‑calling capabilities <<chunk_7,chunk_8>>.

In practice, you can start with a model that supports function‑calling (e.g., GPT‑4), define a catalog of tools (REST endpoints, Python snippets, etc.), optionally fine‑tune it on a tool‑use instruction set like ToolBench, and augment inference with a retrieval component that supplies the latest API docs. This combination enables the LLM to **understand user intent, select the appropriate external tool, construct a valid request, invoke the tool, and incorporate the result** into a coherent answer.

Large‑language models can be extended with “tool‑calling” so that, instead of trying to answer everything from their internal parameters, they invoke external services (APIs, calculators, browsers, etc.) to obtain up‑to‑date or computational results.  The literature describes several practical ways to enable this capability:

1. **Built‑in plugins / official tool APIs** –  
   Leading LLM providers have already added plugin interfaces that let the model call external APIs through a standardized function‑call mechanism.  For example, OpenAI’s ChatGPT (gpt‑3.5‑turbo‑16k) can be upgraded with function‑call support, and GPT‑4 has a dedicated plugin system that can invoke web‑search, calculators, translation services, and more <<chunk_1,chunk_2>>.

2. **Explicit tool‑use prompting** –  
   When the model is given a prompt that describes a desired tool (e.g., “use a web‑browser to look up the latest stock price”), it can generate a structured API call in the response.  Early work demonstrated this with a small, well‑documented set of tools such as web‑browsing, calculators, translation systems, and Python interpreters <<chunk_2>>.

3. **Tool‑specific fine‑tuning (ToolBench)** –  
   A systematic pipeline can be built that first collects a large catalog of real‑world REST APIs (e.g., 16 464 APIs from RapidAPI across 49 categories) and then automatically generates instruction‑tuning data (ToolBench) by prompting a strong LLM (ChatGPT) to write diverse, multi‑tool instructions for those APIs <<chunk_3,chunk_6>>.  Training on this data teaches the model to understand API documentation, select the appropriate endpoint, and fill in required parameters.

4. **Retriever‑aware training** –  
   To handle ever‑changing API docs, the training data can prepend the retrieved API specification (as JSON) to the user query.  The model learns to parse the second half (the doc) and use it to answer the first half (the user request), which improves zero‑shot tool use, reduces hallucinations, and lets the model adapt to new or updated APIs at test time <<chunk_7>>.

5. **Multi‑tool composition** –  
   Real‑world tasks often require chaining several tools (e.g., fetching weather data, then booking a hotel, then sending a confirmation email).  The instruction‑generation process deliberately samples combinations of APIs and asks the model to produce a single plan that invokes multiple services, thereby training the model to reason across steps and pass intermediate results from one tool to the next <<chunk_6>>.

6. **Evaluation frameworks** –  
   Benchmarks such as ToolBench, APIBench, and ToolQA provide large collections of real API calls and responses, enabling systematic evaluation of a model’s ability to generate correct calls, handle multi‑step reasoning, and avoid hallucinations.  These datasets also include metrics for multi‑tool scenarios and retrieval‑augmented reasoning <<chunk_8>>.

In practice, you can start by enabling the built‑in function‑call or plugin interface of a hosted LLM, then progressively move to a custom fine‑tuned model trained on a curated, multi‑tool instruction set (like ToolBench) and augmented with retrieval of up‑to‑date API documentation.  This combination gives you both the flexibility of calling any REST endpoint and the robustness that comes from targeted training and evaluation.

Tool calling with large language models (LLMs) can be approached in several complementary ways:

1. **Integrating pre‑built plugins or extensions** – Major LLM providers have begun to ship plugins that let the model invoke external APIs (e.g., web search, calculators, translation services, Python interpreters) through a standardized API‑calling interface <<chunk_1,chunk_10>>.  

2. **Using a curated repository of real‑world APIs** – Projects such as ToolBench collect thousands of REST APIs (e.g., from RapidAPI) and provide the associated documentation, parameters, and example calls so that an LLM can learn to generate and execute genuine API requests <<chunk_3,chunk_6>>.  

3. **Prompt‑based tool selection** – By prompting the LLM with a description of the desired functionality, the model can select an appropriate tool from a known set and produce the exact code or HTTP request needed to call it (e.g., “convert spoken audio to text using Torch Hub”) <<chunk_4>>.  

4. **Retriever‑aware training** – During instruction tuning, the model is given the retrieved API documentation together with the user query, teaching it to parse the documentation and produce a correct API call even when the underlying API changes at test time <<chunk_7>>.  

5. **Multi‑tool composition** – Instruction datasets can be built to require the LLM to combine several tools in a single workflow (e.g., fetch weather data, then book a hotel, then send a confirmation email), which improves the model’s ability to handle realistic, multi‑step scenarios <<chunk_2,chunk_6,chunk_8>>.  

6. **Fine‑tuning on tool‑use data** – Systematic fine‑tuning on large, high‑quality tool‑use corpora (such as ToolBench) helps the model learn the syntax, parameter constraints, and error handling needed for reliable tool invocation <<chunk_7>>.  

7. **Response compression and error filtering** – After an API call, the raw response can be compressed to fit the LLM’s context window, and low‑quality or failing APIs can be filtered out during dataset construction, ensuring that the model only learns from reliable tool interactions <<chunk_9>>.  

By combining these strategies—plugin integration, curated API collections, prompt engineering, retrieval‑augmented training, multi‑tool composition, fine‑tuning, and robust data preprocessing—developers can enable LLMs to call external tools effectively across a wide range of applications.

3. **ToolBench / ToolLLM framework** – A systematic pipeline for tool use has been built that (a) gathers thousands of high‑quality APIs, (b) automatically generates diverse instruction‑tuning data (including multi‑tool scenarios), and (c) fine‑tunes the LLM on this data.  This approach enables the model to handle both single‑tool and multi‑tool queries and to reason about constraints such as parameter size or accuracy <<chunk_3,chunk_6,chunk_7>>.

This combination lets the LLM act as a natural‑language interface to a wide range of computational tools.

For a univariate random variable the Gaussian (normal) density is  

\[
p(x \mid \mu,\sigma^{2})=\frac{1}{\sqrt{2\pi\sigma^{2}}}\,
\exp\!\left[-\frac{(x-\mu)^{2}}{2\sigma^{2}}\right].
\]  

For a multivariate random variable \(\mathbf{x}\in\mathbb{R}^{D}\) with mean vector \(\boldsymbol{\mu}\) and covariance matrix \(\Sigma\), the density is  

\[
p(\mathbf{x}\mid\boldsymbol{\mu},\Sigma)=
(2\pi)^{-D/2}\,|\Sigma|^{-1/2}\,
\exp\!\left[-\frac12(\mathbf{x}-\boldsymbol{\mu})^{\top}\Sigma^{-1}(\mathbf{x}-\boldsymbol{\mu})\right].
